## Abstract 

Transformer-based LLMs achieve strong results on many language tasks; however, long inputs remain challenging because context windows are finite and prefill latency and memory grow rapidly with prompt length. Flat token-stream processing and chunk-based retrieval can therefore spend substantial computation and con-text budget on text unrelated to the query. Offline-indexed RAG additionally introduces external storage and index management overhead, and typically appends retrieved evidence as raw text, increasing prefill cost and latency. H 2MT makes long-context infer-ence structure-aware: it builds a semantic hierarchy offline, com-putes a memory embedding for each node via bottom-up post-order aggregation, and routes queries coarse-to-fine at inference to prune irrelevant branches early. On LongBench QA (NarrativeQA, Hot-potQA, QASPER) and structured technical-document QA, H 2MT achieves favorable quality–efficiency trade-offs: it substantially re-duces TTFT and often peak GPU memory relative to memory-token baselines, while trading some lexical accuracy against text-retrieval baselines for lower total generation time and no raw-text concate-nation at inference. 

## Keywords 

long-context inference, memory tokens, hierarchical routing, effi-cient LLM inference 

## 1 Introduction 

Natural language processing applications increasingly rely on large language models to process long documents, often structured (e.g., scientific papers or technical manuals), for downstream tasks such as question answering. Since the introduction of the Transformer [ 17 ], self-attention has served as the core primitive behind most modern LLMs. However, dense self-attention scales quadratically with sequence length, making long inputs expensive in both com-pute and memory on common hardware. Consequently, deployed models typically expose finite context windows, which further con-strains long-document processing [10, 22]. A large body of work addresses these bottlenecks by modifying attention or introducing auxiliary mechanisms to extend the effec-tive context, including sparse attention patterns [ 1 , 5 , 23 ], low-rank or linearized attention approximations [ 6, 18 , 25 ], segment-level re-currence [ 3 , 7 ], and external memory mechanisms for long-context efficiency, such as kNN retrieval over cached representations [ 21 ]. While these approaches reduce long-context overhead, they of-ten require non-trivial architectural modifications or heavyweight auxiliary systems, which can alter model behavior and introduce additional hyperparameters and failure modes. Moreover, even with such mechanisms, LLMs can struggle to reliably use evidence that is far from the query [14]. Beyond efficiency concerns, many long-context methods treat documents as flat token sequences. This is a mismatch for real-world artifacts, where the document’s semantic structure (sections, subsections, and scoped definitions) provides a strong inductive bias for locating and composing evidence. Ignoring this structure can spread computation across distant and irrelevant content, par-ticularly in technical documents with localized terminology and hierarchical dependencies. Memory-token approaches (e.g., ICAE, HMT, R 3Mem) compress long contexts into latent memory repre-sentations, but they typically operate over linear segment streams rather than explicitly leveraging document hierarchy for coarse-to-fine routing and controllable evidence granularity at inference [9, 12, 19]. We propose H 2MT, a hierarchy-aware memory framework that preserves a document’s semantic hierarchy and shifts long-range context conditioning from token-level attention to conditioned memory composition. H 2MT constructs a semantic hierarchy over the context using available document metadata (e.g., a table of con-tents for manuals) or by segmenting text into coherent units. It then computes a single memory embedding per node, a fixed-size latent representation that distills the node’s local content and descendant information via bottom-up aggregation. At inference time, H 2MT performs coarse-to-fine routing from the root, prunes low-scoring subtrees early, and conditions generation only on selected node memories. The design adds only lightweight pooling and projection layers on top of a frozen backbone, yielding a parameter-efficient mechanism for long, structured documents. Our contributions are as follows. (1) We introduce H 2MT, a plug-in hierarchy-aware memory interface that represents a long docu-ment as a rooted semantic tree and conditions a frozen backbone on a small set of selected node memories rather than raw long context, without modifying the backbone attention blocks. (2) We develop an offline hierarchical memory construction procedure that com-putes a fixed-size memory embedding for every node via bottom-up post-order aggregation, enabling reusable cached node memories across queries. ## 2 Related Work 

Prior work on long-context inference and retrieval for language models spans (i) structure-aware architectures that encode docu-ment hierarchy, (ii) memory-augmented and other context-efficient mechanisms that reduce the overhead of conditioning on long his-tories, and (iii) retrieval-based methods that reorganize external corpora, sometimes hierarchically, to surface relevant evidence under tight context budgets. 

Structure-aware long-document models. The Hierarchical Docu-ment Transformer (HDT) [ 11 ] injects document structure into the attention pattern via auxiliary anchor tokens (e.g., [SENT], [SEC], [DOC]) and a sample-dependent, multi-level sparse attention kernel, complemented by hierarchical positional encodings. HDT enables structured communication among siblings and parent–child nodes within a document tree while maintaining sparsity. More broadly, structure-aware designs impose document hierarchy through mod-ified attention masks or positional schemes, shaping how infor-mation flows across sections and paragraphs. These methods are effective, but they typically require modifying the backbone’s at-tention pattern or positional encoding scheme, whereas we target a plug-in interface on top of a frozen backbone. 

Memory-augmented recurrence and external memory. The Hierar-chical Memory Transformer (HMT) [ 12 ] provides a model-agnostic memory interface that compresses each processed segment into memory embeddings and retrieves relevant memories during subse-quent processing. HMT organizes memories into multiple tiers and implements sequence-recurrent processing, reporting improved long-context inference quality and reduced inference-time memory in long-context settings. External memory can also be implemented as retrieval over cached representations: Memorizing Transformers [21 ] augment attention with kNN lookup over a key–value store of past activations to extend effective context beyond the raw token window.

Sliding-window attention limits long-range interactions, and exten-sions such as Longformer [ 2] add global interactions to enlarge the effective receptive field. Related methods approximate attention to reduce cost (e.g., Performer [ 6]) or introduce recurrence (e.g., Transformer-XL [ 7], RMT [ 3 ]). A complementary direction trains explicit compressors that map long prefixes to shorter represen-tations for a frozen backbone. ICAE [ 9 ] learns synthetic tokens that encode long histories, and R 3Mem [ 19 ] uses reversible com-pression to balance retention and retrieval. Prompt compression methods such as LongLLMLingua [ 13 ] reduce prompt length by rewriting or dropping less salient spans. These approaches improve long-context efficiency, but they typically operate over linear to-ken or segment streams rather than leveraging native document hierarchy (e.g., sections and subsections) for coarse-to-fine routing and controllable evidence granularity. 

Hierarchical retrieval for retrieval-augmented generation. RAP-TOR [ 16 ] builds a multi-level index over a corpus using iterative clustering and LLM-based abstraction, enabling retrieval at multiple granularities rather than flat top-𝑘 chunk selection. At inference time, it can retrieve nodes from different levels under a token bud-get, mixing higher-level abstractions with lower-level details. This hierarchy is induced over an external corpus, whereas our focus is exploiting a document’s internal semantic structure and routing over cached node memories during inference. 

## 3 Methodology 

H2MT consists of three stages: (i) constructing the semantic hierar-chy tree of the context (Section 3.1), (ii) constructing hierarchical memory tokens for all nodes in the tree, including intermediate nodes (Section 3.2), and (iii) hierarchical memory-aware inference (Section 3.3). The design targets long-context settings while keeping the backbone architecture unchanged.

## 3.3 Hierarchical Memory-Aware Inference 

Given a query q about a long context and a precomputed semantic hierarchy T = (𝑉 , 𝐸, 𝑟 ), inference performs coarse-to-fine localiza-tion over the tree using the node memories {𝑚 𝑣 ∈ R𝑑 : 𝑣 ∈ 𝑉 }.The goal is to retrieve the set of relevant subtrees by using the bottom-up memories computed offline and to speed up inference. The intuition aligns with a common pattern in human recall: when people are asked a question about previously memorized material, they typically form a high-level outline first, use it to localize the relevant part of their memory, and only then retrieve the fine-grained details within that region. Without this coarse-to-fine localization, accessing specific details is often slower and more error-prone.

## 4 Experiments 4.1 Setup 

Datasets We evaluate on five long-context QA datasets. Three are benchmarks from LongBench (QASPER, HotpotQA, NarrativeQA), and two are technical-document QA sets (OpenROAD and a VLSI circuit layout verification manual which is related to one of the leading electronic design automation vendors). QASPER uses the full text of a research paper as the context, with questions targeting paper-specific details such as methods and ex-perimental settings. HotpotQA is multi-document QA, where each example provides multiple documents and the answer require evi-dence across them. NarrativeQA is single-document QA grounded in long narratives (stories or scripts), requiring reasoning about characters, plot, etc. OpenROAD and the VLSI circuit layout verification manual eval-uate the same capability in structured technical manuals. For Open-ROAD, we use ORD-QA, a documentation QA benchmark intro-duced by Pu et al . [15] , consisting of question–document–answer triplets grounded in the OpenROAD documentation, which is open-source EDA tool. The other manual is a private QA set constructed from a a set of VLSI circuit layout verification manuals from a major electronic design automation company; we treat the manual text as the corpus and derive question–answer pairs using our internal generation pipeline. We describe the QA construction and quality control procedure in the appendix. 

Preprocessing We construct a semantic hierarchy out of each long context. The construction is domain-specific: for technical documentation (OpenROAD, VLSI layout verification manual), the hierarchy follows the table of contents and based on this metadata we can find the relation between different parts of the context and generate the semnatic hierarchy. For NarrativeQA, each book or script is organized as document to chapter or scene to paragraph leaves (with an optional configurable minimum paragraph counts in each node); for HotpotQA, each Wikipedia title is a root node and its sentences are depth-1 leaves. For QASPER, the hierarchy follows paper to section to paragraph leaves. Table 5 summarizes node statistics of the semantic hierarchy for each dataset. 

Baselines We compare H 2MT against both model-side long-context methods and prompt-compression baselines. HMT [ 12 ]leverages memory-augmented segment-level recurrence to orga-nize the memory hierarchy by preserving tokens from early input segments, passing memory embeddings along the sequence, and recalling relevant information from history.. ICAE [ 9] trains an in-context autoencoder that compresses a long context into a small set of latent memory slots that the backbone conditions on for genera-tion. MemoryLLM [ 20 ] augments the model with a fixed-size latent memory pool and supports memory-only conditioning; we also report its retrieval-assisted variant (e.g., BM25-filtered memories). UltraGist is represented by the Activation Beacon line [ 24 ], which compresses long-context computation via compact “beacon” repre-sentations that the model conditions on downstream. For input-side compression, LongLLMLingua [ 13 ] and AutoCompressors [ 4] re-duce the effective prompt length by selecting or rewriting salient content, while leaving the backbone architecture unchanged.

TTFT vs. peak GPU memory. TTFT and peak GPU memory capture different bottlenecks and do not necessarily move together. TTFT is dominated by prefill latency up to the first generated to-ken, while peak memory reflects the maximum footprint over the entire run, often dominated by KV-cache growth during decoding or by temporary buffers. As a result, reducing TTFT by shortening the effective prefill can still increase peak memory when routing retrieves more nodes (larger 𝐾 ) or when deeper traversal increases the number of memory tokens inserted.

## 4.3 GMM-induced hierarchy for HotpotQA 

HotpotQA does not provide an intrinsic document hierarchy and because of this we see low performance, so we induce a seman-tic tree by clustering chunk-level memory representations. Our construction borrows the bottom-up, recursive hierarchy-building principle from RAPTOR, however, with memory tokens not by raw texts. We first split each context document into chunks and encode each chunk using our MemoryCell, producing a memory represen-tation for every chunk. We then fit a Gaussian Mixture Model over these chunk memories to obtain clusters, and create a parent node per cluster by aggregating the memories of its assigned children. We repeat this procedure on the newly created parent memories to build higher levels of the tree until the process stops due to insuffi-cient compression or reaching a maximum depth. This produces a hierarchy over the HotpotQA context, which we use at inference time for top-down routing and memory selection. The table 7 highlights a quality–efficiency trade-off between H2MT (GMM) and SBERT+RAPTOR. The ROUGE gap is consis-tent with the fact that our method is lossy by design : H 2MT com-presses raw text into memory representations and further aggre-gates them during clustering and parent construction, which can discard token-level evidence that overlap-based metrics reward. ## 4.4 Retrieval-Augmented Generation Baseline 

Since our setting requires query-conditioned selection from a static long document, we additionally compare against retrieval-augmented generation (RAG) baselines. We consider RAPTOR [ 16 ], which builds a hierarchical index by recursively embedding and cluster-ing text chunks, and generating abstractive summaries for clusters using an LLM. At inference time, RAPTOR retrieves nodes from multiple abstraction levels and concatenates their raw texts (inter-nal summaries and, when selected, leaf-level chunks) as context for the generator. Empirically, SBERT+RAPTOR is a strong accuracy baseline (Ta-ble 7). It is also important to note that RAPTOR conditions the generator on the explicit text of retrieved nodes (including internal summaries and leaf chunks), whereas H 2MT routes over fixed-size learned memory tokens and never concatenates raw retrieved spans at inference. This difference makes H 2MT intentionally lossy : com-pressing each node into a bounded representation (and applying hard top-𝑘 selection) can discard fine-grained evidence needed for multi-hop reasoning, and any hierarchy construction errors (e.g., imperfect GMM clustering or parent assignment) are difficult to recover from at inference because the model attends to compressed memories rather than token-level text. In contrast, RAPTOR pre-serves more explicit evidence through retrieved text and summaries, which can improve overlap-based accuracy metrics, but typically increases both offline index construction cost and online genera-tion time due to longer effective context and additional decoding (Table 7).

## 5 Conclusion 

We presented H 2MT, a plug-in hierarchy-aware memory frame-work for long-document question answering that preserves docu-ment structure while keeping the backbone architecture unchanged. H2MT constructs a semantic hierarchy tree offline, builds a single cached memory embedding per node via bottom-up aggregation, and performs online coarse-to-fine routing with per-parent prun-ing to select a small set of relevant node memories for generation. Across LongBench QA and structured technical-document settings (OpenROAD), our results show that H 2MT can improve generation quality while substantially reducing time-to-first-token and often peak GPU memory compared to strong long-context baselines. 

Limitations. A key limitation is that H 2MT relies on a meaning-ful document hierarchy. For corpora with weak or noisy structure (like HotpotQA), hierarchy induction is currently heuristic rather than fully automatic, and errors in the induced tree can propagate to memory construction and routing decisions. In addition, com-pressing an entire subtree into a single memory vector at each internal node can attenuate fine-grained signals: relevant evidence H2MT 

may be present in a descendant, yet its contribution can be diluted at higher levels, causing the corresponding branch to be pruned during top-down routing. 

Future Work. Future work includes improving automatic hier-archy induction for weakly structured corpora, incorporating richer structures beyond trees (e.g., graphs capturing cross-references and shared entities), and learning routing and aggregation policies that better preserve rare but critical evidence while optimizing accu-racy–latency–memory trade-offs.

## A LVS dataset and QA construction 

LVS is a set of VLSI circuit layout verification manuals from a major electronic design automation company. We derive the LVS-QA set directly from the product manual’s HTML corpus. Each HTML file is typed as rich or light using simple structural cues. We mark a file as 

rich if it contains <pre> or <code> blocks, tables with header rows, “Warning/Note/Caution” callouts, option matrices, or numbered constraint lists; otherwise it is labeled light . We reconstruct the document hierarchy from the manual’s table of contents (TOC) and define a file’s related context as its immediate children in this tree. We instruct an open-source generator (DeepSeek-R1 [ 8]) to pro-duce one to 𝑁 question–answer pairs answerable from the file context. To match file type, we restrict question categories: 

• Rich: definition, usage, syntax, parameter, warning, con-straint, list_children, where_to_find_child, code_example. 

• Light: definition, scope, usage, list_children, syntax. By construction, each QA item is grounded in the union of a node and its immediately related files in the TOC hierarchy.
