## 摘要
长上下文LLM推理面临一个根本冲突：头自适应压缩算法（例如Top-$p$核采样）通过动态波动内存预算提供卓越的准确性，然而现代推理引擎（如vLLM）需要刚性、静态的内存模式以利用CUDA Graphs和PagedAttention。我们通过HARD-KV解决这种“静态-动态”不匹配，这是一个统一框架，将动态选择与刚性系统约束连接起来。HARD-KV引入了一个级联缓存层次结构，管理token在密集层、稀疏层和压缩层的生命周期。关键的是，我们提出了一个Logits校准机制，将多样化的重要性指标归一化为统一的概率空间，从而实现跨异构头的持续Top-$p$预算分配。为了弥合效率差距，我们提供了一个系统级解决方案，将碎片化的动态索引重写为与高性能推理引擎兼容的连续物理布局。在数学推理基准（AIME, U-Math）上的大量实验验证了，HARD-KV在10k+ token场景下实现了高达2倍的吞吐量提升，同时保持了高保真生成。代码可在https://github.com/SuDIS-ZJU/HARDInfer获取。

## Abstract
Long-context LLM inference faces a fundamental conflict: head-adaptive compression algorithms (e.g., Top-$p$ nucleus sampling) offer superior accuracy by dynamically fluctuating memory budgets, yet modern inference engines (e.g., vLLM) demand rigid, static memory patterns to leverage CUDA Graphs and PagedAttention. We resolve this ``Static-Dynamic'' mismatch with HARD-KV, a unified framework that that bridges dynamic selection with rigid system constraints. HARD-KV introduces a Cascade Cache hierarchy, managing the token lifecycle across dense, sparse, and condensed tiers. Crucially, we propose a Logits Calibration mechanism that normalizes diverse importance metrics into a unified probability space, enabling consistent Top-$p$ budgeting across heterogeneous heads. To bridge the efficiency gap, we offer a system-level solution, which rewrites fragmented, dynamic indices into contiguous physical layouts compatible with high-performance inference engine. Extensive experiments on math-reasoning benchmarks (AIME, U-Math) verify that HARD-KV achieves up to 2$\times$ throughput improvement over static baselines while maintaining high-fidelity generation in 10k+ token scenarios. Code is available at https://github.com/SuDIS-ZJU/HARDInfer.

## 2. 论文提出的方法论：核心思想、关键技术细节

### 2.1 整体架构：Cascade Cache（级联缓存）

- **三层存储层次**：
  - **Tier 1: Dense Cache**（密集缓存）：缓存最近 tokens 的连续内存，保留注意力池（attention sinks）和局部句法依赖，实现全保真局部注意力。
  - **Tier 2: Sparse Cache**（稀疏缓存）：token 移出密集窗口后进入稀疏缓存，应用**校准后的Top‑p选择**（见2.2）保留信息密集的 KV 对，实现头自适应预算分配。
  - **Tier 3: Condensed Cache**（压缩缓存）：最旧的块进一步通过驱逐或加权合并压缩成固定大小存储，满足硬内存限制。

### 2.2 Logits Calibration（Logits校准）机制

- **问题**：不同 KV 选择方法（如 SnapKV 的 max‑pool、R‑KV 的 key 相似度）产生的重要性度量尺度不一，直接应用 Top‑p 会导致**选择坍塌**（如图2所示，某些方法在相同 p 下保留 token 数量急剧减少）。
- **统一化框架**：将不同方法抽象为三步：(1) Gather（按子组聚合），(2) Reduce‑Scatter（计算代理评分并散布），(3) Sample（Top‑k 或 Top‑p 采样）。在第二步后引入温度缩放校准。
- **保持顺序不变性**：若变换函数 g(z) 保持排序，则使用 `ˆS = S(g(z/T))`；否则使用 `ˆS = S(g(z)/T)`。通过梯度下降或二分搜索为每个注意力头/层寻找最优温度 T，使 Top‑k 集合的累积概率质量与原始注意力分布相近。
- **效果**：校准后，不同方法在统一 p 下产生相似的 token 选择密度和模式（如图3所示的热力图），使 Top‑p 预算能动态适应注意力分布。

### 2.3 系统级索引正则化

- **三种操作**：
  - **Rewrite Cache**：将非连续选中的 KV 块读并写入连续物理块，保证后续解码步骤的索引一致性。
  - **Sparse Loading**：在 kernel 执行时通过硬件注意力掩码实现层‑头动态加载，但可能冗余加载未选中块。
  - **Flatten Index**：将各头的块索引展平为一维，按最大长度或总和统一长度。
- **策略选择**：根据不同配置（Token‑wise、Layer‑wise、Head‑wise）选择组合，文中重点讨论 Head‑wise Allocation (HA) 场景，并给出 HA‑Sparse 和 HA‑Max 两种方案。HA‑Max 在长序列下通过重写换取更快的计算（如图5所示，10k+ token 时计算延迟减半）。
- **Three Principles**：P1: 联合优化内存占用和延迟；P2: 使用头‑独立块索引支持 CUDA Graphs；P3: 稀疏度无关管理（总稀疏度影响大于最大稀疏度）。

### 2.4 谨慎更新策略（Cautious Update）

- 在 Tier1→Tier2 过渡时，将密集窗口遍历视为“投票期”，维护累积选择掩码，仅在块完全离开密集窗口后压缩，防止临时不重要的 token 被过早驱逐。
