## 摘要
LLM服务正日益被来自智能体、推理模型和扩展对话的长时间动态解码工作负载所主导。当突发性长上下文需求超出部署容量时，现有服务系统通常通过启动额外的服务实例并复制模型来进行横向扩展。这种实例级弹性只能通过提供另一份完整的模型副本来增加KV容量，这会带来启动延迟、内存开销和批处理碎片问题。我们提出了KernelFlume，一种以解码为中心的架构，它将稳定的投影/FFN路径与核心注意力计算分离：权重节点执行密集的投影/FFN内核，而无权重注意力节点存储令牌范围的KV分区，并根据请求状态需求进行扩展。为了使这种分离具有弹性，KernelFlume维护一个路由表，将令牌范围映射到注意力节点端点。它在令牌边界更新路由，并使用主机可见的图形信号驱动预先注册的UCX端点通信（在捕获的CUDA图之外）。为了在分离后保持低每令牌延迟，KernelFlume将查询优先的核心注意力调度与层间内核流水线相结合，将远程注意力和通信与本地投影/FFN工作重叠。在真实GPU测试平台（节点内A6000和跨节点H100）上，针对服务Llama-3.1-8B的动态长上下文智能体工作负载，KernelFlume在A6000上维持约74毫秒、在H100上维持约34毫秒的稳定p99 TPOT，同时相对于使用最先进的实例启动方法ServerlessLLM的全实例弹性扩展，每百万输出令牌的成本分别降低了高达32%和61%。在更大模型规模下重放相同轨迹的仿真显示，相比ServerlessLLM成本降低56%至66%，在使用更便宜的异构注意力节点硬件时，成本降低幅度扩大至80%至85%，并持续到百万令牌上下文范围。

## Abstract
LLM serving is increasingly dominated by long and dynamic decode workloads from agents, reasoning models, and extended conversations. When bursty long-context demand exceeds deployed capacity, existing serving systems typically scale out by launching additional serving instances with model replicas. This instance-level elasticity increases KV capacity only by provisioning another full copy of the model, inheriting startup latency, memory overhead, and batch fragmentation.   We present KernelFlume, a decode-centric architecture that disaggregates the stable projection/FFN path from core-attention computation: weight nodes execute dense projection/FFN kernels, while weightless attention nodes store token-range KV partitions and scale with request-state demand. To make this separation elastic, KernelFlume maintains a routing table that maps token ranges to attention-node endpoints. It updates routes at token boundaries and uses host-visible graph signals to drive pre-registered UCX endpoint communication outside the captured CUDA Graph. To preserve low per-token latency after disaggregation, KernelFlume combines query-first core-attention dispatch with inter-layer kernel pipelining, overlapping remote attention and communication with local projection/FFN work. On real GPU testbeds (intra-node A6000 and cross-node H100), under a dynamic long-context agentic workload serving Llama-3.1-8B, KernelFlume sustains flat p99 TPOTs of ~74 ms on A6000 and ~34 ms on H100, while lowering cost per million output tokens by up to 32% and 61%, respectively, relative to full-instance elastic scaling with ServerlessLLM, a state-of-the-art instance-startup method. Replaying the same trace at larger model scale in simulation projects a 56--66% cost reduction over ServerlessLLM, widening to 80--85% with cheaper heterogeneous attention-node hardware and persisting into the million-token context range.

## 1. 核心问题与整体含义（研究动机和背景）

- **背景**：LLM服务正从单轮对话转向长时间、动态的解码工作负载，如深度推理、自主智能体（agent）和多轮交互。这些工作负载的显著特征是：KV缓存需求随上下文（sequence length）持续增长，但模型权重保持不变；同时请求具有高动态性，序列长度和KV增长不可预测，突发性长上下文需求经常超出部署容量。
- **核心问题**：现有弹性扩展方案（如实例级scale-out）存在三大局限：
  - **高启动开销**：加载新实例时需复制完整模型权重，即使使用优化方法（如ServerlessLLM），启动时间仍远超解码SLO（服务级别目标，如TPOT要求）。
  - **低内存效率**：每个扩展步必须携带完整模型权重副本（如Llama-3.1-8B的18 GB），浪费了本该用于KV缓存的HBM。
  - **低计算效率**：将请求分散到多个实例会缩小每实例的有效batch size，降低GEMM利用率。
- **动机**：需要一种更细粒度的弹性扩展方式，即**只扩展KV容量而不复制模型权重**，从而避免上述问题。

## 2. 方法论：核心思想、关键技术细节

### 2.1 核心思想：解耦投影/FFN与注意力
- **物理分拆**：将模型解码过程拆分为两类节点：
  - **权重节点（Weight Node, W节点）**：持有模型权重，负责执行QKV投影、FFN等计算密集型操作，保持大batch高利用率。
  - **注意力节点（Attention Node, A节点）**：无权重（weightless），仅存储KV缓存分区，执行本地注意力计算（core attention）。这些节点可动态增删，以适应KV需求变化。
- **弹性单位**：从“复制整个实例”变为“增删一个无权重A节点”，避免权重重复。

### 2.2 关键技术细节
- **弹性路由控制平面**：
  - 使用**路由表**将token范围（如[0,50K) → A0, [50K,100K) → A1）映射到UCX端点句柄。
  - 路由更新在token边界进行，不修改CUDA Graph、不重建通信器、不暂停进行中的解码步。
  - 通信基于UCX（点对点传输），接收端使用预发布的GPU缓冲区，实现零拷贝。
  - 通过主机可见信号（WriteValue32）通知CPU路由循环，CPU根据路由表发起UCX发送。
- **图进度信号机制**：CUDA Graph只包含本地计算和信号写入，不嵌入通信目标；CPU轮询信号并查询路由表后发起传输，使路由更新不影响图的重放（graph replay）。
- **弹性缩放策略（Algorithm 1）**：
  - 根据HBM容量和TPOT SLO，定义每个A节点的KV容量阈值 \(C_{\text{thresh}} = \min(\beta_{\text{mem}} C_{\text{kv\_cap}}, \beta_{\text{slo}} T_{\text{max}}^A BW_{\text{HBM}})\)。
  - 预填充后，根据总KV占用 \(C_{\text{total}}\) 计算所需A节点数 \(N_A = \lceil C_{\text{total}} / C_{\text{thresh}} \rceil\)。
  - 预测性扩容：对于每个尾节点 \(i\)，估计剩余填充时间 \(\tau_{\text{fill}}^{(i)} = (C_{\text{thresh}} - b_i) / (n_i \cdot 2H_{\text{kv}} d_h L \cdot 2B)\)。当剩余时间不足以在后台完成预热时，启动新A节点预热，并在达到水位线后于下一个token边界发布路由。
- **查询优先注意力分派（Query-First Attention, QFA）**：
  - 将Q投影与K/V投影分离：一旦Q产生，立即将其通过路由表发送到所有持有历史KV的A节点，使这些A节点在W节点继续K/V投影的同时开始注意力计算，从而重叠通信与计算。
  - 尾节点需等待(K_new, V_new)到达后再更新本地KV并计算注意力。
  - 每个A节点返回局部部分结果（o_i, m_i, ℓ_i），W节点使用在线softmax合并。
- **层间内核流水线（Inter-layer Kernel Pipelining）**：
  - 将解码请求按微批次（microbatch）划分，例如M=2。
  - W节点在一个微批次上执行投影和FFN的同时，A节点在另一个微批次上计算注意力，使得每层暴露时间变为 \(\max(T_p + T_f, T_A(N_A))\)，理想情况下可达到2×加速比（当W侧与A侧时间接近时）。

## 6. 主要结论与发现

- **弹性扩展效率**：KernelFlume通过增删无权重A节点，实现线性KV容量增长而TPOT几乎平坦（~72ms on A6000），拓扑切换仅在临界路径引入7.2μs（预预热后）。
- **QFA与流水线有效性**：
  - QFA将分离开销从最高11%降低到<5%（MHA模型甚至比非分离更快）。
  - 内核流水线达到1.87×–1.94×的加速比，接近2倍上界，模型预测误差3.1%。
- **端到端成本降低**：
  - 在真实测试平台上，相比ServerlessLLM，KernelFlume降低$/Mtok 21%–32%（A6000）和27%–61%（H100），同时保持100% SLO达成率。
  - 仿真中，Llama-70B规模降低56%–66%，异构硬件（H20作A节点）可扩大至80%–85%，效果持续至百万token上下文。
- **核心发现**：将弹性单位从“完整实例”改为“无权重KV节点”，可以将容量变化从重量级实例管理转变为轻量级路由更新，同时改善尾延迟和服务成本。

## 7. 优点

1. **创新性架构**：首次在解码阶段内部分离投影/FFN与注意力，使KV容量扩展不再绑定权重复制。
2. **极低切换开销**：通过路由表+图外信号机制，弹性事件仅需7.2μs临界路径开销，远低于实例级方案（秒级）。
3. **高效重叠执行**：查询优先分派和层间流水线有效掩盖远程通信和计算延迟，使分离后性能接近甚至优于非分离版本。
4. **弹性策略完善**：同时考虑内存和延迟约束，预测性预热避免突发溢出。
5. **多硬件验证**：在A6000和H100两种不同架构、不同互联的平台上均取得一致优势。
6. **可扩展性**：仿真表明方法可推广到更大模型（70B/405B）和更长上下文（百万token），且异构硬件可进一步降本。

## 8. 不足与局限

1. **实验覆盖范围**：
   - 仅使用了单一轨迹（Codex/SWE-bench Pro），缺乏其他类型长上下文工作负载（如多轮对话、文档摘要）的验证。
   - 未考虑MoE模型（如DeepSeek）或混合专家架构，这些模型可能需要不同的分离策略。
2. **硬件依赖**：目前仅验证了NVIDIA GPU（A6000/H100/H20），对AMD、Intel等平台未涉及。
3. **仿真精度风险**：70B/405B仿真依赖硬件参数模型，虽经8B校准，但实际部署中可能因内存带宽波动、通信争用等因素产生偏差。
4. **未解决KV迁移问题**：当A节点退役时，现有KV需等待请求完成才能释放，未讨论主动迁移策略以优化负载均衡。
5. **缺少训练/预填充优化**：论文聚焦解码弹性，但预填充阶段仍可采用传统方法（如chunked prefill），未讨论预填充与KernelFlume的互动。
6. **对SLO严格性要求高**：弹性策略依赖准确的时间预测和稳定的HBM带宽，若出现严重干扰（如其他进程争抢带宽），可能导致SLO违规。
