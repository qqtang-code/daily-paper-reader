## 摘要
多模态大语言模型（MLLMs）最近通过下一个词元预测直接建模查询-文档相关性，显示出作为点式重排序器的强大潜力。然而，点式重排序在查询-文档对之间遭受大量重复计算，而Transformer的因果结构仅允许通过预缓存重用前缀段。为了解决现有的查询优先和文档优先格式与VQA风格提示及计算感知重用之间的不匹配，我们提出了一种\textit{视觉优先}的公式，提高了缓存重用效率和重排序性能。然而，剩余成本仍然相当可观，主要来自三个来源：（1）\textit{模型深度}，我们通过提前退出减少活跃参数；（2）\textit{跨段注意力}，我们将其限制在少数层中的狭窄交互带内；（3）\textit{视觉词元}，我们通过嵌入器引导的剪枝减少词元数量。这些设计共同构成了miniReranker，在单一查询的高重用设置下，将重排序运行时间减少到密集实现的<1%，同时保持了>96%的密集模型性能。

## Abstract
Multimodal large language models (MLLMs) have recently shown strong potential as point-wise rerankers by directly modeling query--document relevance through next-token prediction. However, point-wise reranking suffers from substantial repeated computation across query--document pairs, while the causal structure of transformers allows only prefix segments to be reused via pre-caching. To address the misalignment of existing query-first and document-first formats with both VQA-style prompting and computation-aware reuse, we propose a \textit{vision-first} formulation that improves both cache reuse efficiency and reranking performance. However, the remaining cost is still considerable and stems from three main sources: (1) \textit{model depth}, for which we reduce active parameters via early exit; (2) \textit{cross-segment attention}, which we restrict to a narrow interaction band across a few layers; and (3) \textit{visual tokens}, where we reduce the number of tokens via embedder-guided pruning. Together, these designs form miniReranker, which reduces reranking runtime to <1% of the dense implementation under high-reuse settings for a single query, while preserving >96% of the dense model performance.

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：多模态大语言模型（MLLMs）作为点式（point-wise）重排序器时，每个查询需与大量候选文档独立配对，导致高度重叠的多模态输入被反复计算，计算成本极高。
- **背景**：
  - 现有MLLM重排序器普遍继承文本检索的 **查询优先（query-first）** 格式（`[ins, q, d]`），但该设计对多模态重排序存在双重瓶颈：
    - **有效性**：MLLM预训练于VQA数据（视觉在前），查询优先格式破坏了原生输入对齐，尤其文本查询→视觉文档场景。
    - **效率**：因果Transformer仅允许前缀KV缓存复用；查询优先迫使视觉文档（通常计算量巨大）被重复编码K次（K为候选数）。文档优先（document-first）又会导致视觉查询（V→T场景）被重复编码。
  - 即使采用最优的序列顺序，MLLM内部仍存在深度、跨段注意力、视觉token三个层面的冗余。

## 2. 方法论：核心思想与关键技术细节

### 2.1 总体框架：miniReranker
- 基于Qwen3-VL构建，融合**视觉优先提示**与三种模型级压缩策略。

### 2.2 视觉优先提示（Vision-First Prompt Reformulation）
- **动态序列排序**：严格将视觉模态置于文本之前：
  - T→V（文本查询，视觉文档）：采用 **文档优先** `[ins, d, q]` → 视觉文档可离线缓存，且恢复VQA格式。
  - V→T（视觉查询，文本文档）：采用 **查询优先** `[ins, q, d]` → 视觉查询编码一次并广播。
  - V→V（视觉查询，视觉文档）：采用文档优先（性能与查询优先相近，但支持文档侧预缓存）。
- **收益**：对齐MLLM预训练格式，同时最大化昂贵的视觉计算复用。理论FLOPs节省显著（例如T→V中消除K次视觉文档在线编码）。

### 2.3 模型级压缩（Model-Level Compression）

#### 2.3.1 早期退出（Early Exit）
- **动机**：通过logit lens分析，重排序准确性在中间层（约60%深度）已达到95%最终层性能，深层主要用于语言精细化和生成。
- **操作**：前向传播在指定中间层停止（2B模型于第16层，4B/8B于第21层），避免深层计算。
- **效果**：活跃参数降至原始模型的58%（2B：1.14B；4B：2.2B；8B：4.6B）。

#### 2.3.2 交互带（Interaction Band）
- **动机**：分析表明有效查询-文档交互仅集中在少数中间层；浅层各段独立演化，深层交互对最终token影响微弱。
- **操作**：限制跨段注意力仅发生在**交互带**内的层（2B：8–16层；4B/8B：8–21层）。训练时使用稀疏注意力掩码，推理时仅需加载交互层中第一段的KV缓存。
- **效果**：消除非交互层中的二次方注意力计算，保留必要跨段融合。

#### 2.3.3 嵌入器引导的Token剪枝（Embedder-Guided Token Pruning）
- **动机**：视觉token数量占序列长度主导，许多token对最终相关性决策贡献有限。
- **操作**：复用上游第一阶段检索器（如Qwen3-VL-Embedding）的计算——聚合所有层的注意力权重（从最终序列token到视觉token），选取top α%（实验中α=50%）的视觉token保留，其余丢弃。
- **优势**：零额外前向开销，且层聚合比单层更鲁棒。

## 6. 主要结论与发现
1. **视觉优先提示**在重排序有效性和效率上均优于查询优先和文档优先，且对微调后模型仍然成立。
2. 点式重排序中存在显著的**深度、交互、token三层冗余**，可通过早期退出、交互带、嵌入器引导剪枝分别缓解。
3. **miniReranker**在保持>96%（2B）和>98%（4B/8B）密集模型性能的同时，将活跃参数降至58%，训练加速~3×。
4. 在高复用场景（如单查询重排序Top-100候选）下，**视频重排序运行时降至密集实现的<1%**，图像运行时降至<15%；且优势随候选数和查询数增加而放大。

## 7. 优点
- **问题洞察深刻**：系统性分析MLLM重排序中的冗余源（序列顺序、深度、注意力、token），并分别提出正交优化。
- **方法论巧妙**：视觉优先提示同时解决对齐和效率问题；嵌入器引导剪枝零成本利用已有计算；交互带精确定位关键融合层。
- **实验扎实**：涵盖78个多模态任务，与9种基线全面对比，消融实验彻底（包括多种token选择策略和组件组合），效率分析维度丰富。
- **工程实用性**：所有压缩策略不引入额外模型，可无缝集成已有检索系统；代码和配置公开（文中未明确，但基于开源模型）。

## 8. 不足与局限
- **点式重排序局限**：仅针对独立评估的point-wise公式，未探索list-wise或agentic检索场景，这些场景可能获得更丰富的跨候选交互信息，但复杂度更高。
- **依赖第一阶段检索器**：token剪枝使用上游embedder的注意力，若第一阶段检索器质量差，可能引入偏差或信息丢失。
- **视觉token剪枝的适应性**：固定保留50%的剪枝率可能不是所有模态和任务的最优值，未讨论自适应剪枝策略。
- **长视频/长文档扩展**：实验虽包含视频，但视频帧采样有上限（64帧，1FPS），对于更长视频可能压缩策略需调整。
- **计算资源消耗**：训练仍需多GPU（4×A100/H100），对于资源受限场景仍有门槛。
- **未讨论系统级部署细节**：如与不同类型检索器配合的兼容性，缓存管理和清空策略等。
