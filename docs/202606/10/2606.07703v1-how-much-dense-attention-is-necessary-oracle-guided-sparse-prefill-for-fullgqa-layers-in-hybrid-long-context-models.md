## 摘要
长上下文预填充仍然昂贵，因为即使在具有局部、稀疏、线性或循环组件的混合模型中，全/GQA层仍会对历史序列进行评分。我们研究了在明确支持粒度（support granularity）和top-k预算下，需要多少密集注意力来保留任务级行为。我们为现有的GQA检查点引入了一种注意力质量top-k oracle：对于每个层和查询位置，它计算密集注意力，选择头部平均的token支持，并仅在该支持上重新计算注意力。该oracle是一个诊断参考，而非可部署的加速器，并将稀疏预算可行性（sparse-budget feasibility）与索引器错误及运行时实现效应分离开来。在Qwen系列检索密集型评估中，每个查询最长的oracle行与密集式保持在1分以内，而Qwen3.5-9B从4K到100K的RULER风格扫描保持在0.48分以内。在oracle的指导下，我们推导出一个头部折叠的辅助索引器，该索引器通过从密集注意力质量分布进行KL蒸馏训练，同时保持骨干网络冻结。使用分别蒸馏的Qwen3.5-0.8B和Qwen3.5-9B索引器，报告的16K/32K验证宏差距分别为+2.04和+1.13分，这被视为质量保留而非改进；融合的选择块共享支持可能会引入更大的实现差距。初步的单卡TTFT测量显示，与密集的FlashAttention-2基线相比，蒸馏索引器稀疏服务的加速比为Qwen3.5-0.8B在NPU上1.71倍，Qwen3.5-9B在GPU上1.93倍。额外的随机初始化压力行达到3.44倍，表明稀疏运行时存在提升空间，但未验证输出质量。本次初次发布将oracle可行性、蒸馏索引器质量和运行时提升空间分离开来，将完全匹配的质量-延迟前沿留待未来工作。

## Abstract
Long-context prefill remains expensive because full/GQA layers still score the historical sequence, even in hybrid models with local, sparse, linear, or recurrent components. We study how much dense attention is needed to preserve task-level behavior under explicit support granularity and top-k budgets.   We introduce an attention-mass top-k oracle for existing GQA checkpoints: for each layer and query position, it computes dense attention, selects head-averaged token support, and recomputes attention only on that support. The oracle is a diagnostic reference, not a deployable accelerator, and separates sparse-budget feasibility from indexer error and runtime realization effects.   On Qwen-family retrieval-heavy evaluations, the longest per-query oracle rows stay within 1 point of dense, and a Qwen3.5-9B RULER-style sweep from 4K to 100K stays within 0.48 points. Guided by the oracle, we derive a head-collapsed auxiliary indexer trained by KL distillation from dense attention-mass distributions while keeping the backbone frozen. With separately distilled Qwen3.5-0.8B and Qwen3.5-9B indexers, the reported 16K/32K validation macro gaps are +2.04 and +1.13 points, treated as quality preservation rather than improvement; fused selection-block-shared support can introduce a larger realization gap.   Preliminary single-card TTFT measurements show distilled-indexer sparse serving speedups of 1.71x for Qwen3.5-0.8B on NPU and 1.93x for Qwen3.5-9B on GPU against its dense FlashAttention-2 baseline. Additional random-init stress rows reach 3.44x, indicating sparse-runtime headroom but not validated output quality. This first release separates oracle feasibility, distilled-indexer quality, and runtime headroom, leaving a fully matched quality-latency frontier to future work.

## 1. 核心问题与整体含义（研究动机和背景）
*   **研究动机**：长上下文模型的预填充（prefill）阶段依然昂贵，因为即使混合架构（如局部、稀疏、线性、循环组件）中，全注意力或分组查询注意力（GQA）层仍需对所有历史序列进行密集评分。这导致首令牌时延（TTFT）随上下文长度急剧增加。
*   **核心问题**：在给定支持粒度（per-query 或 selection-block-shared）和 top-k 预算下，“需要多少密集注意力才能保留任务级行为？”即能否通过少量关键token的稀疏注意力替代全量密集注意力，同时保持输出质量。
*   **整体含义**：论文提出了一个**oracle引导的稀疏预填充框架**，旨在隔离影响稀疏加速的三个误差源：oracle gap（稀疏预算本身导致的损失）、indexer gap（蒸馏索引器预测支持不准确导致的损失）、realization gap（运行时实现如块共享支持导致的额外损失）。通过分离这些因素，更系统地评估稀疏预填充的可行性和优化方向。

## 2. 方法论：核心思想与关键技术细节
*   **核心思想**：先利用一个**注意力质量 top-k oracle**（不部署，仅作诊断）确定在给定预算下密集注意力的支持是否足够稀疏；然后基于oracle的密集注意力分布，通过KL蒸馏训练一个轻量级**头部折叠索引器**，使其能预测相近的稀疏支持；最后在冻结骨干模型的前提下，用该索引器指导预填充阶段的稀疏注意力计算。
*   **关键技术细节**：
    *   **注意力质量 top-k oracle**：对每层每个查询位置，计算完整的密集注意力，将各查询头平均得到 token 重要性分布 \( P_{\ell,t}(s) \)；选取该分布中 top-k token 作为支持，然后仅在这些 token 上重新计算稀疏注意力。支持粒度可以是 per-query（每个查询独立选支持）或 selection-block-shared（bsel=64个连续查询共享一个支持集）。
    *   **头部折叠蒸馏索引器**：保留骨干模型冻结，额外训练一个单头索引器。其输入是隐藏状态，输出每个 token 的得分 \( z_{\ell,t}(s) \)；通过最小化与 oracle 分布 \( P_{\ell,t} \) 的 KL 散度来训练。索引器使用 ReLU 激活和 RoPE 编码，并通过头平均（collapsed）设计减少辅助参数。推理时，该索引器根据得分选择 top-k 支持，然后原始模型仅在该支持上执行 GQA 计算（解码阶段仍保持密集注意力）。
    *   **算法流程**：
        1.  使用序列中的隐藏状态通过索引器得到得分。
        2.  对得分应用因果掩码，选择 top-k token 索引。
        3.  将选中的支持广播到所有 KV 头。
        4.  在原始 Q、K、V 上，仅对选中支持执行稀疏 GQA 预填充。
*   **公式/算法**（文字说明）：\( P_{\ell,t}(s) = \frac{1}{H_q}\sum_{h=1}^{H_q} A_{\ell,t,h,s} \)，其中 \( A_{\ell,t,h,s} \) 是密集注意力权重。索引器训练目标：\( L_{KL} = \frac{1}{|Q|}\sum_{(\ell,t)\in Q} KL(P_{\ell,t} \| \text{softmax}(z_{\ell,t})) \)，其中 \( Q \) 是训练样本中的层和位置集合。

## 3. 实验设计：数据集、基准与对比方法

    *   **Oracle 实验**：使用非公开的 RULER 风格长上下文评估（13个检索密集型任务：代码词提取 CWE、频繁词提取 FWE、Needle-in-a-Haystack 变体、QA、变量跟踪 VT）。长度覆盖 4K 到 128K（部分达 256K）。

## 6. 主要结论与发现
1.  **GQA 层的密集注意力可高度稀疏化**：在 Qwen 家族检索密集任务上，per-query oracle 在 top-k=2048（或恰当安排）下，质量损失 <1 分；Qwen3.5-9B 从 4K 到 100K 平均差距仅 0.48 分。表明密集注意力质量集中在少数 token 上。
2.  **支持共享需要长度感知的预算**：selection-block-shared（bsel=64）在长上下文（如128K）中若预算过低（如 top-k=256）会导致严重性能下降（检索任务“悬崖”）。合理的预算需随长度增加（如32K需512，64K需1024，128K需2048）。
3.  **蒸馏索引器可保持质量并实现加速**：使用蒸馏索引器的稀疏预填充（密集解码）在 16K/32K 验证集上宏平均差距分别 +2.04分（0.8B）和 +1.13分（9B），视为质量保持；TTFT 加速在 NPU 上 1.71×，GPU 上 1.93×；随机初始化压力测试显示最高 3.44× 运行时头潜力。
4.  **误差分离**：oracle gap 较小，indexer gap 和 realization gap 是主要质量损失来源（特别是 fused 路径）。未来需进一步缩小这些 gap。

## 7. 优点：方法与实验设计的亮点
*   **方法创新**：提出“oracle引导”框架，将稀疏预填充的研究从“直接尝试”转变为“先诊断后优化”，通过三大 gap 清晰定位瓶颈。
*   **非侵入式**：保持骨干冻结，仅添加 auxiliary indexer，无需修改主模型，便于集成到现有系统。
*   **灵活性**：索引器训练后支持可调 top-k 预算，适应不同精度和延迟需求。
*   **系统分离**：明确区分 oracle sparse（可行性）、distilled sparse（实际质量）、random-init sparse（系统头）三种证据类型，避免混淆部署效果。
*   **在多个 Qwen 模型上系统评估**：覆盖 0.6B、0.8B、8B、9B、27B、35B-A3B，长度至 256K，验证了方法的鲁棒性。

## 8. 不足与局限
*   **模型家族局限**：实验仅聚焦 Qwen 系列，未在 Gemma、Jamba 等其他混合架构上验证。方法对于不同注意力布局（如局部+全局配置）的泛用性未知。
*   **缺少与其他稀疏系统的对比**：没有与 MInference、DSA、NSA 等现有方法进行质量-延迟公平比较，无法判断本方法的实际优势。
*   **质量-延迟匹配不完整**：TTFT 加速测量（如 128K/256K）缺乏对应长度的质量验证（质量仅在 16K/32K 报告）。长上下文下的索引器精度未知。
*   **统计信息缺失**：未报告置信区间、重复性方差或 bootstrap 结果，单次测量的可靠性存疑。
*   **可重复性低**：数据集、检查点、代码、原始日志未公开，基于内部数据，难以复现。
*   **消融不足**：头部折叠 vs. 头部扩展索引器没有消融；不同 $d_{idx}$、训练采样策略的影响未充分讨论。
*   **Oracle 本身昂贵**：oracle 计算需要完整密集注意力，无法直接部署，仅作诊断工具。实际部署仍需蒸馏索引器引入额外开销和误差。
