## 摘要
前沿大语言模型越来越多地使用稀疏注意力索引器来决定查询关注的内容，该索引器为每个查询选取少量 KV 缓存块：注意力的单位现在是一个小的、可复用的块。代理型工作负载大量使用这一机制：许多子代理查询一个大型代码库，重复使用相同的块。当该语料库超过单个 GPU 容量时，会被分区到多个实例上，因此查询及其选中的块通常位于不同的 GPU 上：回答查询意味着跨实例的注意力。先前跨实例 KV 系统的惯常做法是移动缓存：将选中的块拉取到请求方。多头潜在注意力（MLA）反转了这一算力逻辑，将每个令牌的键和值压缩为一个窄向量，因此一个路由后的查询行仅约 1 KB，比它关注的块还要小；此时路由查询往往比移动缓存更便宜。哪种原语在何种结构和请求形态下占优，目前尚无定论，尤其是在设备发起的 RDMA 上，它使得每个请求的跨节点传输成本低廉。我们在真实的多节点 H100 集群上对跨实例 MLA 注意力进行了特征分析，提炼出两个可复用的成果：一个拓扑感知的成本模型（探测/传输/计算/返回/合并）和一个闭式的路由/获取/本地谓词，其常数我们在真实 IBGDA 上测量得到，该模型跟踪批量往返的误差在约 7% 以内。在解码阶段，它路由查询，以移动缓存（对于一个连续块约为 3 毫秒的重新适应拼接，或在选择下的分散收集）的代价换取数十微秒的往返，并通过探测延迟而非峰值带宽来选择结构。我们为 MLA 实例化了成本模型和谓词，但两者并非 MLA 特有：它们适用于任何压缩或稀疏选择将注意力缩小到小块的场景（如今如 DeepSeek-V3.2、V4 和 GLM-5.1）。将其扩展到新架构只需测量两个系数：路由有效载荷和获取的移动缓存成本。

## Abstract
Frontier LLMs increasingly decide what a query attends to with a sparse-attention indexer that picks a few KV-cache blocks per query: attention's unit is now a small, reusable chunk. Agentic workloads hammer it: many sub-agents query one large codebase, reusing the same blocks. When that corpus outgrows one GPU it is partitioned across instances, so a query and the blocks it selects often sit on different GPUs: answering it means attention across instances. The reflex of prior cross-instance KV systems is to move the cache: pull the selected blocks to the requester. Multi-head Latent Attention inverts the arithmetic, compressing each token's key and value into one narrow vector, so a routed query row is only ~1 KB, smaller than the chunk it attends; routing the query is then often cheaper than moving the cache. Which primitive wins, over which fabric and request shape, is uncharted, least of all on device-initiated RDMA that makes per-request cross-node transfers cheap. We characterize cross-instance MLA attention on a real multi-node H100 cluster, distilling two reusable artifacts: a topology-aware cost model (probe / transfer / compute / return / merge) and a closed-form route/fetch/local predicate, whose constants we measure on real IBGDA, where the model tracks batched round-trips to within ~7%. At decode it routes the query, trading the cost of moving the cache (a ~3 ms re-adaptation splice for a contiguous chunk, or a scattered gather under selection) for a tens-of-microsecond round trip, and picks the fabric by probe latency, not peak bandwidth. We instantiate the cost model and predicate for MLA, but neither is MLA-specific: they apply wherever compression or sparse selection shrinks attention to small chunks (DeepSeek-V3.2, V4, and GLM-5.1 today). Extending them to a new architecture requires measuring just two coefficients: the routed payload and fetch's move-the-cache cost.

## 论文总结：Move the Query, Not the Cache

### 1. 论文的核心问题与整体含义（研究动机和背景）

- **问题背景**：在大语言模型（LLM）推理中，跨实例注意力（cross-instance attention）频繁出现，尤其是当共享的规范语料库（如代码库、法律文档）超过单个 GPU 的 HBM 时，KV 缓存被分区到多个 GPU。查询（query）与其选中的 KV 缓存块常位于不同 GPU，传统做法是**移动缓存**（fetch）——将远程块拉到查询方实例，再本地做注意力计算——这带来毫秒级的位置适配拼接（~3 ms splice）或分散收集开销。
- **核心问题**：多头潜在注意力（MLA）将每个 token 的键和值压缩为窄向量（约 1 KB/行），使得路由查询（route）变得比移动缓存更便宜？哪种原语在哪种 fabric 和请求形状下占优？设备发起的 RDMA（IBGDA）是否使路由可行？
- **整体含义**：路由查询将移动缓存的毫秒级开销转化为数十微秒的往返，从根本上改变了跨实例注意力的成本结构，使分布式注意力成为可扩展的选择。该发现适用于所有压缩或稀疏选择注意力架构（如 DeepSeek-V3.2、V4、GLM-5.1）。

### 2. 论文提出的方法论：核心思想、关键技术细节

- **核心思想**：利用 MLA 的窄潜在表示（query row ≈ 1 KB），将查询路由到持有缓存块的实例，在该实例上计算部分注意力，返回部分统计量（输出向量 + log-sum-exp 统计），查询方通过在线 softmax 合并部分结果。这避免了移动整个缓存块（数 MB）及其位置适配开销。
- **关键技术细节**：
  - **路由原语**：查询方发送 batch of query rows（$b\times d_{qk}$），接收方返回每个请求的部分注意力结果（$partial = (o, m, \ell)$）。通过设备发起的 RDMA（NVSHMEM IBGDA）实现单块 put+signal 的微秒级往返。
  - **成本模型**：拓扑感知的五项分解模型：
    \[
    T_{\text{redist}}(F, s, B) = T_{\text{probe}}(F) + T_{\text{transfer}}(F, s, B) + T_{\text{compute}} + T_{\text{return}}(F, s, B') + T_{\text{merge}}
    \]
    其中 $F$ 为 fabric（NVLink、IBGDA、PCIe），$s$ 为设备端范围（thread/warp/block），$B$ 为传输单元大小，$T_{\text{probe}}$ 为探测延迟，$T_{\text{transfer}}$ 与字节数线性相关，$T_{\text{compute}}$ 为持有方的部分注意力计算时间，$T_{\text{merge}}$ 为查询方的在线 softmax 合并。
  - **闭式选择谓词**：对于给定的请求和 chunk，系统比较三种原语的估算成本，选取最小值。核心为：
    - route: $T_{\text{route}} = T_{\text{probe}} + \frac{M_q(q+p)}{BW}$
    - fetch: $T_{\text{fetch}} = T_{\text{pull}} + T_{\text{splice}}$
    - local: $T_{\text{local}} = T_{\text{prefill}}(c_t)$
  - **正确性验证**：路由-合并结果与单实例注意力在 bf16 噪声内一致（max abs < 0.0014），在稀疏选择场景下同样验证。

- **对比方法**：

  - **fetch**（移动缓存）：传统做法，含位置适配拼接（splice）和拉取所有层的 KV 缓存。

### 6. 论文的主要结论与发现

- **路由查询在解码阶段显著优于移动缓存**：路由往返约 31–48 μs（batch=256），而 fetch 至少 ~3 ms 拼接 + 数据拉取，优势 60–100 倍。
- **成本模型准确**：线性模型预测批量路由延迟的 MAPE 为 7%（batch≥512），常数随 fabric 变化但结构通用。
- **路由适用于所有压缩/稀疏注意力**：模型输入仅两个系数（路由 payload 和 fetch 的移动缓存成本），可扩展到 DeepSeek-V3.2/V4、GLM-5.1。
- **稀疏选择下路由优势更大**：分散收集 (gather) 延迟随持有方数量线性增长，路由 fan-out 平坦。
- **fabric 选择由 probe 延迟而非峰值带宽决定**：解码批量太小，无法饱和高带宽链路，路由成本相近（31–48 μs 在五种 fabric 上）。
- **拥塞下路由仍远优于 fetch**：即使完全饱和链路，路由延迟仍低于 splice 约 10 倍。
- **实践规则**：解码默认路由；fetch 仅用于多次复用的 chunk；local 仅用于极小 chunk（<75–220 tokens）；holder 并发流控制在 8 条。

- **成本模型结构化**：五阶段分解清晰，常数易测量，可迁移至新架构。
- **实测驱动**：在真实 H100+IBGDA 集群上验证，而非模拟。
- **正确性验证严格**：路由-合并与原注意力几乎位一致，并验证了稀疏选择下的准确性。
- **充分考虑 fabric 和拓扑影响**：涵盖 5 种 fabric、同 leaf/跨 spine、内部 socket/跨 socket。

### 8. 不足与局限

- **仅微基准，无完整 serving 系统评估**：未与生产 serving 系统（如 vLLM、SGLang）集成验证端到端性能。
- **主机开销尚未消除**：当前原型中主机开销（如 Python 客户端、调度）使路由在 batch 小时反而不如无拼接的 bytes-back fetch，论文认可是工程问题但未实现。
- **绝对延迟受驱动版本限制**：使用遗核闭源驱动，可能低估 IBGDA 性能，但结论相对优势稳定。
- **未评估 FP8 注意力**：论文使用 bf16 MLA kernel，生产环境常用 FP8，但作者认为 FP8 只会缩小稀疏 kernel 开销，不影响结论。

- **仅测量单块 put+signal**：多块并行可提高带宽，但作者认为解码时路由已足够快，无需；仅在极端大 batch 有用。
- **持有方计算仅测量解码 kernel**：未评估 prefill 或稀疏 kernel 的混合场景。
